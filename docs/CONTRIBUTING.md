# Contributing to Dmstic

**Author**: Dariusz Porczyński
**Updated**: 2026-03-28
**Project**: Dmstic — Analiza kosztów domowych (Household Cost Analytics)

Thank you for your interest in contributing to Dmstic. This document covers the development workflow, code style, commit format, and how to submit changes.

---

## Getting Started

### 1. Fork and clone

```bash
git clone https://github.com/Dmstic/dmstic.git
cd dmstic
```

### 2. Set up a local development environment

Requirements:
- PHP 8.3+
- Composer 2.x
- MySQL 8.x or MariaDB 10.6+
- Git

```bash
composer install
cp .env.example .env
php artisan key:generate
```

Configure `.env` with your local MySQL credentials, then:

```bash
php artisan migrate
php artisan storage:link
php artisan serve
```

Application available at `http://localhost:8000`.

### 3. Seed with test data (optional)

```bash
php artisan db:seed
```

Or add a provider manually via `http://localhost:8000/admin/provider/create` and upload a sample invoice.

## Development Workflow

1. Create a feature branch from `main`:
   ```bash
   git checkout -b feat/your-feature-name
   ```

2. Make your changes. Follow the code style guidelines below.

3. Run tests:
   ```bash
   php artisan test
   ```

4. Commit using the commit format below.

5. Push and open a Pull Request against `main`.

---

## Code Style

### PHP

- Follow **PSR-12** coding standard
- Use PHP 8.3 features where they improve clarity: `match`, `readonly`, named arguments, enums
- Type declarations on all function parameters and return types
- No unused variables, no commented-out code in PRs

**Compact format (project rule)**:
- Minimal empty lines within functions — keep code dense
- Inline comments preferred: add at end of the line, not as a separate line above
- No multiline comment blocks (except file/class docblocks)

```php
// BAD
function calculateCostPerKwh(float $gross, float $kwh): ?float
{
    // Check for division by zero
    if ($kwh <= 0) {
        return null;
    }

    // Calculate and return
    return round($gross / $kwh, 4);
}

// GOOD
function calculateCostPerKwh(float $gross, float $kwh): ?float
{
    if ($kwh <= 0) { return null; }  // Guard: no division by zero
    return round($gross / $kwh, 4);
}
```

### Blade Templates

- Use `@` directives consistently (`@foreach`, `@if`, `@isset`)
- No raw PHP in views (`<?php` blocks) — use view composers or controller data
- Escape output with `{{ }}` (not `{!! !!}`) unless HTML output is explicitly required and trusted

### JavaScript

- Vanilla JS preferred — no additional JS frameworks (Chart.js is already included)
- Use `const` and `let`, never `var`
- Inline scripts in Blade files acceptable for small Chart.js configurations
- Larger JS logic should go in `public/js/`

### CSS

- CSS custom properties for theming: `--primary`, `--background`, `--text`, etc.
- Avoid inline styles in HTML
- Dark/light theme via `[data-theme="dark"]` attribute on `<html>` element

## Commit Format

All commits must follow this format:

```
emoji type: short description
```

### Emoji by type

| Emoji | Type | When to use |
|-------|------|-------------|
| ✨ | feat | New feature |
| 🐛 | fix | Bug fix |
| ♻️ | refactor | Code restructuring without behaviour change |
| 📝 | docs | Documentation only |
| 🎨 | style | Formatting, whitespace, CSS |
| ✅ | test | Adding or fixing tests |
| ⚙️ | config | Configuration files |
| 🗄️ | db | Database migrations or schema changes |
| 🚀 | deploy | Deployment or infrastructure changes |
| 🔒 | security | Security improvements |
| ⬆️ | deps | Dependency updates |
| 🗑️ | remove | Removing code or files |

### Examples

```
✨ feat: add year-over-year pivot table to provider page
🐛 fix: cost-per-kwh division by zero when kwh is null
♻️ refactor: extract monthly aggregation into ProviderService
📝 docs: add TAURON setup instructions to README
🎨 style: improve sidebar dark theme contrast
✅ test: add feature test for settings page
🗄️ db: add index on bills.issue_date for filter performance
```

### Rules

- **One line only** — no multiline commit messages
- **No links** — no URLs in commit messages
- **No AI attribution** — do not include "generated by", "Claude", "AI-assisted", etc.
- **No co-author lines** — commits appear as author's work only
- **Present tense** — "add feature" not "added feature"
- **Lowercase** after the colon (except proper nouns)

## Pull Request Guidelines

### PR title

Same format as a commit message: `emoji type: short description`

### PR description

Include:
- What the change does
- Why it is needed (link to an issue if applicable)
- How to test it manually
- Any database migrations required

### Checklist before opening a PR

- [ ] Tests pass: `php artisan test`
- [ ] No debug code (`dd()`, `dump()`, `var_dump()`) left in
- [ ] Migrations are reversible (`down()` method implemented)
- [ ] `.env.example` updated if new env variables added
- [ ] Docs updated if user-facing behaviour changed

## Adding a New Provider

To add support for a new utility provider:

1. Create a scraper class in `app/Services/Scrapers/`:
   ```php
   class YourProviderScraper
   {
       public function __construct(private array $credentials) {}
       public function fetchDocuments(): array { /* ... */ }
       public function downloadPdf(string $docId): ?string { /* ... */ }
   }
   ```

2. Add a record to `energy_providers` via admin UI or migration.

3. Map the scraper class name in the provider's `scraper_class` field.

4. Document the provider's API in `docs/PROVIDERS.md` (if the file exists), or add notes to the PR.

5. Ensure credentials fields are documented in `credentials.json.example` (in the private repo — coordinate with the maintainer).

## Reporting Issues

Open a GitHub Issue with:
- Description of the problem
- Steps to reproduce
- Expected vs actual behaviour
- PHP version, Laravel version, OS

For security vulnerabilities: do not open a public issue. Contact the maintainer directly.
